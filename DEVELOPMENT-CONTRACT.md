# Development Contract — Agent Runtime & Backend

This document defines the **hard architectural contract** between the agent runtime and this backend repository. It is non-negotiable. Every code change, PR, and architectural decision inside this repo must conform to the boundaries defined here. Violating this contract is architectural regression.

---

## 1. Backend Authority Boundaries

The backend is **not** an orchestration participant. It is a deterministic acquisition substrate.

### The backend is NO LONGER ALLOWED to:

| Prohibited | Reason |
|---|---|
| Own orchestration timing | Orchestration belongs to the Supervisor only |
| Own synchronization cadence | Sync cadence is computed by the SyncLayer |
| Own freshness logic | Freshness is a runtime cognition concern |
| Directly wake runtime execution | Wake authority belongs to the Supervisor |
| Interpret strategic state | Strategy is bounded cognition, not infrastructure |
| Expose arbitrary fetch endpoints | All acquisition MUST originate from AcquisitionIntents |
| Run autonomous synchronization loops | No timer-driven sync, no cron-based acquisition |
| Embed orchestration reasoning | No hidden retry semantics tied to orchestration state |

### The backend EXISTS SOLELY to:

- **Acquire** — interact with Instagram and external APIs
- **Validate** — sanitize and validate external state
- **Normalize** — transform raw responses into canonical schemas
- **Persist** — write normalized state to canonical DB
- **Emit telemetry** — publish acquisition results and health signals

**One sentence governs all backend decisions:** _The backend is a deterministic acquisition substrate._

## 2. Runtime vs Backend Responsibility Split

| Concern | Owner | Rationale |
|---|---|---|
| What context is needed | Runtime (Supervisor) | Strategic evaluation |
| Why synchronization matters | Runtime (Supervisor) | Coherence computation |
| When acquisition is justified | Runtime (Supervisor) | Deterministic threshold evaluation |
| How orchestration proceeds | Runtime (Supervisor) | Lifecycle governance |
| Acquire external data | Backend | Transport execution |
| Validate response integrity | Backend | Normalization boundary |
| Normalize to canonical schema | Backend | Persistence contract |
| Persist to canonical DB | Backend | State mutation |
| Emit telemetry | Backend | Observability signal |

**The runtime owns _meaning_. The backend owns _transport and normalization_.** This separation is critical and must never blur.

## 3. AcquisitionIntent Contract

This is the **ONLY** synchronization interface between the agent runtime and the backend.

The backend must never expose arbitrary "fetch endpoints" that the runtime calls freely. All acquisition originates from deterministic `AcquisitionIntent` records generated by the supervisor-governed SyncLayer.

### Hard Invariant Flow

```
Supervisor
    |
SyncLayer
    |
AcquisitionIntent (canonical record)
    |
Redis Queue (intent delivery)
    |
AcquisitionWorker (backend consumer)
    |
Backend Acquisition (transport execution)
    |
Canonical DB Persistence (normalized write)
    |
Realtime Telemetry (mutation signal)
    |
Runtime Reevaluation (supervisor loop)
```

### AcquisitionIntent Schema (canonical)

```json
{
  "intent_id": "uuid",
  "intent_type": "analytics | engagement | messages | moderation | publishing | attribution",
  "target_resource": "instagram_user_id or post_id etc.",
  "parameters": {},
  "priority": "low | normal | high | critical",
  "idempotency_key": "uuid",
  "issued_by": "sync_layer",
  "issued_at": "ISO8601",
  "expires_at": "ISO8601"
}
```

### AcquisitionResult Schema (canonical)

```json
{
  "result_id": "uuid",
  "intent_id": "uuid (foreign key)",
  "status": "success | partial | failure | timeout",
  "normalized_payload": {},
  "acquisition_latency_ms": 123,
  "retry_count": 0,
  "error_code": "string | null",
  "completed_at": "ISO8601"
}
```

**Any code that bypasses this contract and directly fetches external state is a violation.**

## 4. Database-Mediated Runtime Flow

This is non-negotiable. The runtime **never** consumes transient backend responses directly.

### The only valid data flow:

1. Backend acquires external data
2. Backend normalizes and writes to canonical DB (Supabase)
3. Supabase emits mutation telemetry (realtime channel)
4. Supervisor reevaluates runtime state from persisted data
5. Cognition consumes reconciled persistence

### This prevents:

- Response coupling between runtime and backend
- Synchronization races from transient state
- Graph bypassing canonical persistence
- Hidden execution semantics from direct API consumption

**The runtime consumes canonical DB state only. Never transient API responses.**

---

## 5. Bounded Realtime Semantics

Realtime is **telemetry only**. It is NOT:

- A wake system
- A synchronization trigger
- An orchestration mechanism

Realtime mutates **runtime awareness state** — it signals that canonical state changed. It does not and must never trigger acquisition, execution, or orchestration escalation.

Earlier generations of this system were entangled with event-driven orchestration, causing wake storms, recursive graph triggering, duplicate synchronization, and unstable runtime behavior. Realtime authority has been fully centralized back into the supervisor plane.

These are repo-level invariants. Any code violating these laws is invalid by definition.

2. **Only the SyncLayer may authorize acquisition.**

4. **Backend may never own orchestration semantics.**
5. **Realtime is telemetry only.**
6. **Runtime consumes canonical DB state only.**
7. **AcquisitionWorkers are deterministic infrastructure executors.**
8. **Cognition is bounded beneath deterministic governance.**

## 8. Backend Rewrite Sequence

The rewrite must stabilize infrastructure topology before introducing higher-order optimization. Order matters.

| Phase | Action | Outcome |
|---|---|---|
| **1** | Extract pure acquisition primitives | API clients, auth handling, retry logic, normalization, persistence writes isolated from orchestration |
| **2** | Remove all timer-driven synchronization | Eliminate `node-cron` jobs, interval-based sync, autonomous refresh loops |
| **3** | Implement Redis-driven AcquisitionIntent consumption | Backend consumes intents from queue; no self-initiated acquisition |
| **4** | Centralize persistence writes | All writes go through canonical normalization pipelines |
| **5** | Implement observability & acquisition telemetry | Structured telemetry, execution lineage, mutation tracing |
| **6** | Split acquisition domains into bounded modules | Reorganize into domain-bounded `acquisition/` modules |

### Immediate anti-patterns to remove from current codebase:

- `node-cron` scheduled jobs (timer-driven synchronization)
- Direct Instagram API calls from route handlers (bypasses AcquisitionIntent)
- Services that mix orchestration logic with transport logic
- Any code path where a route handler independently decides _when_ to fetch

## 10. Frontend-Facing Backend Concerns

The backend serves **two distinct consumers** that must never be conflated:

| Consumer | Owner of _what context is needed_ | Backend role |
|---|---|---|
| **Agent Runtime** (Python/LangChain) | Supervisor determines meaning | Deterministic acquisition substrate — the AcquisitionIntent contract applies |
| **Frontend Dashboard** (human operator) | User directly requests data | Transport, normalization, auth, and persistence for human-initiated operations |

**The agent-facing backend and the frontend-facing backend share infrastructure but not contracts.** Code paths that serve the frontend must not inherit the AcquisitionIntent contract. Conversely, AcquisitionIntent-driven code must not be called directly by the frontend.

### Frontend-Facing Backend Scope

The frontend-facing backend handles all user-initiated, human-facing operations:

| Domain | Responsibilities |
|---|---|
| **Authentication** | Supabase OAuth token exchange, JWT verification, session management, credential caching |
| **OAuth Token Lifecycle** | Exchange short-lived tokens for page tokens, store, retrieve, refresh, validate long-lived tokens |
| **Dashboard Data** | Media browsing, inbox read, UGC review, sync status, token health — all consumed by the React dashboard |
| **Webhook Reception** | Receive Meta webhook events (Path B), write to DB for frontend polling, forward to agent (Path A) |
| **Legal / Meta Compliance** | Privacy policy, terms of service, data deletion callback (Meta-required POST endpoint), deletion status |
| **Health & Monitoring** | Health checks, database status, system status — for uptime monitoring |

### Architectural Rule: No Cross-Contamination

- **Shared infrastructure only:** Supabase client, rate limiting middleware, webhook verification, logging — these are neutral.

## 14. Backend-Agent Bridge: Current State

There are currently **two bridge channels** between the backend and the agent runtime. They must be unified into one DB-mediated reactive bridge.

### Channel A — Direct Bypass (ELIMINATE — dangerous)

```
Meta Webhook → backend webhook.js → forwardToAgent() →
  axios.post(AGENT_URL + '/webhook/comment' or '/webhook/dm')
```

- **Type:** Fire-and-forget HTTP POST — no DB write, no persistence, no replay
- **Data:** Normalized webhook event (comment or DM field only)
- **Problem:** If agent is down, event is **permanently lost**. No retry, no replay, no observability
- **Problem:** Agent receives raw HTTP forward — not a structured queue item
- **Problem:** Only handles `comments` and `messages` fields; other webhook types are dropped silently
- **Problem:** Bypasses canonical persistence entirely — violates Law 6

**This is the contagious instability.** One component (`forwardToAgent`) introduces a non-replayable, non-observable path that bypasses every safety mechanism in the stack.

### Channel B — DB-Mediated (current, but polling-based)

```
Agent → POST /agent/heartbeat → agent_heartbeats table
Frontend → GET /agent/status (polls DB)
```

- **Type:** DB write + HTTP polling — correct pattern, wrong delivery mechanism
- **Data:** Heartbeat timestamp, agent liveness
- **Problem:** Frontend polls `/agent/status` — should instead subscribe to Supabase Realtime

### Channel C — Legacy Sync Bridge (ELIMINATE)

```
services/sync/ (cron jobs) → Instagram API → Supabase
```

- **Type:** Backend-owned timer-driven fetch — violates Law 2
- **Problem:** Backend independently decides when to acquire, bypassing AcquisitionIntent contract entirely
- **Problem:** Produces data the agent **consumes** — this is the graph-triggered acquisition pattern that caused earlier generations to destabilize

### Current Bridge Map

```
                    ┌─────────────────────────────────────────────┐
                    │           BACKEND                            │
                    │                                             │
                    │  ┌─ forwardToAgent() ───────────────────────┐│
Meta ──── Webhook ──│  │  DIRECT BYPASS (Channel A) — ELIMINATE   ││
                    │  │  Fire-and-forget HTTP → agent            ││
                    │  │  No DB write. No replay. No observability ││
                    │  └─────────────────────────────────────────┘│
                    │                                             │
                    │  ┌─ heartbeat.js ───────────────────────────┐│
Agent ──────────────│  │  DB-MEDIATED (Channel B) — REACTIVE     ││
                    │  │  POST /agent/heartbeat → agent_heartbeats││
                    │  │  GET /agent/status ← polling (FIX THIS) ││
                    │  └─────────────────────────────────────────┘│
                    │                                             │
                    │  ┌─ services/sync/ ────────────────────────┐│
Backend cron ───────│  │  LEGACY (Channel C) — ELIMINATE         ││
                    │  │  Timer-driven → Instagram API → Supabase ││
```

## 15. Unified Backend-Agent Bridge Design

The backend and agent share **one bridge**: DB writes, with Supabase Realtime as the event delivery mechanism. No HTTP forwards. No fire-and-forget. No polling.

### Unified Bridge

```
Agent Runtime (Supervisor)               Backend                    Supabase
     │                                    │                          │
     │  AcquisitionIntent                 │                          │
     │  (Redis queue) ────────────────────►│                          │
     │                                    │                          │
     │                                    │  writes canonical state  │
     │                                    │  ──────────────────────► │
     │                                    │                          │
     │                                    │              Supabase    │
     │                                    │              Realtime    │
     │                                    │              ◄────────── │
     │  subscribes to postgres_changes ◄──┘                          │
     │                                                                  │
     │  (no polling, no direct HTTP)                                   │
```

### What Changes

| Channel | Current | Replace with |
|---|---|---|
| **Channel A** (forwardToAgent) | Meta webhook → direct HTTP to agent | Meta webhook → DB write (`audit_log`, domain table) → Supabase Realtime → agent subscribes |
| **Channel B** (heartbeat poll) | Agent POSTs heartbeat; frontend polls GET /agent/status | Agent POSTs heartbeat → DB; frontend subscribes to `agent_heartbeats` via Supabase Realtime |
| **Channel C** (cron sync) | Backend cron owns acquisition cadence | AcquisitionIntent-driven workers consume from Redis queue → write DB → Supabase Realtime |

### Concrete Changes Required

**webhook.js:**
1. `forwardToAgent()` must be **removed entirely**
2. Every incoming Meta webhook event must be **written to Supabase** (`audit_log` + relevant domain table)
3. Backend writes canonical state — Supabase Realtime fires — agent subscribes to the relevant `postgres_changes` channel

**heartbeat.js:**
1. Remove `/agent/status` polling endpoint (or mark deprecated)
2. Frontend subscribes to `postgres_changes:agent_heartbeats` for liveness
3. `POST /agent/heartbeat` remains — agent still writes to DB, but delivery is reactive not polled

**services/sync/:**
1. All 6 cron jobs eliminated (see Section 14 Bus 1)
2. Replaced by AcquisitionIntent-driven workers consuming from Redis queue

### Supabase Realtime Channel Scope for Bridge

```
postgres_changes:audit_log
  → Agent subscribes to `webhook_comments` and `webhook_messages` event_type filter
  → Replaces forwardToAgent() completely

postgres_changes:agent_heartbeats
  → Agent runtime (liveness monitoring)
  → Frontend dashboard (reactive liveness indicator, replaces /agent/status polling)

postgres_changes:post_queue
  → Post-fallback worker (reactive queue processing, replaces cron)
  → Agent runtime (sees queue events reactively)
```

## 16. Single Reactive Data Bus Design

The goal is one reactive bus where the backend is purely reactive — it responds to state changes rather than driving them.

### Architecture: Supabase Realtime as the Backbone

```
Agent Runtime (Supervisor)
    │  emits AcquisitionIntent (canonical record)
    ▼
Redis Queue  ←——— Backend AcquisitionWorker (passive consumer)
    │               │
    │               │  acquires from Instagram
    │               ▼
    │         Supabase (canonical persistence)
    │               │
    │               │  Supabase Realtime (postgres_changes)
    │               ▼
    └──────► Agent Runtime (subscribes to postgres_changes)
                  │
                  ▼
            Cognition (reactive to DB mutation)
```

### The Single Bus

**Supabase Realtime `postgres_changes`** becomes the **only** event channel.

- Backend writes canonical state to Supabase
- Supabase emits realtime changes automatically
- Agent runtime subscribes to relevant channels
- No polling, no in-memory cache, no cron-driven acquisition

### What replaces what

| Current mechanism | Replaced by | Notes |
|---|---|---|
| `global.realtimeCache` + polling | Supabase Realtime channel on `audit_log` + domain tables | Frontend subscribes directly to Supabase Realtime |
| `forwardToAgent()` direct bypass | Supabase Realtime channel on `audit_log` | Agent subscribes, no HTTP forward |
| `services/sync/` cron jobs | Supabase Realtime subscription + AcquisitionIntent-triggered workers | Worker is purely reactive to intent queue |
| `GET /webhook/realtime-updates` | Supabase Realtime channel | Eliminated |
| `GET /agent/status` polling | Supabase Realtime on `agent_heartbeats` | Eliminated |
| In-memory `_rateLimitedAccounts` | Supabase table `rate_limit_state` | Persisted, survives restart |
| In-memory `_credentialCache` | Redis (credential cache remains, intent-driven) | 5-min TTL, refreshed on intent |

### Implementation Sequence

| Phase | Action | Outcome |
|---|---|---|
| **R1** | Add first Supabase Realtime subscription to `agent_heartbeats` — replace Bus 4 polling | Agent liveness via realtime, not heartbeat cron |
| **R2** | Add Supabase Realtime subscriptions on `post_queue` — post-fallback consumer becomes reactive to DB changes | Eliminates post-fallback cron, replaced by DB trigger |
| **R3** | Replace `global.realtimeCache` with Supabase Realtime channel on `audit_log` | Frontend subscribes to realtime, no polling |
| **R4** | Replace `forwardToAgent()` — webhook writes to DB, agent subscribes via Supabase Realtime | Direct bypass eliminated permanently |
| **R5** | Eliminate `services/sync/` cron jobs — replace with AcquisitionIntent-driven workers consuming from Redis | AcquisitionIntent contract enforced, no timer-driven sync |
| **R6** | Agent runtime subscribes to Supabase Realtime for all canonical state mutations | Complete reactive loop: intent → acquisition → persistence → realtime → cognition |

### Invariant

**The backend never initiates acquisition. It only responds to:**
1. AcquisitionIntents arriving in the Redis queue (from the agent runtime Supervisor)
2. Supabase Realtime events firing on subscribed tables
3. Direct frontend user actions (OAuth token exchange, manual sync trigger)

**The backend does not poll, cron, or schedule acquisition. The backend does not bypass DB for agent communication.**

The backend currently has **5 distinct data bus mechanisms** that must be collapsed into one reactive bus:

### Bus 1 — Proactive Cron Sync (LEGACY — must be replaced)

```
services/sync/index.js
  ├── engagement.js    — cron */3 * * * * → IG API → instagram_comments / dm_conversations / dm_messages
  ├── ugc.js           — cron 0 */3 * * * → IG API → ugc_content
  ├── media.js         — cron 0 */6 * * * → IG API → instagram_media
  ├── insights.js     — cron 0 2 * * *   → IG API → instagram_media_insights
  ├── token-health.js  — cron 0 3 * * *   → Meta debug API → instagram_credentials (is_active)
  └── helpers.js       — cron */5 * * * * → heartbeat failover → post_queue
```

**Problem:** Timer-driven, backend-owned acquisition cadence. Violates AcquisitionIntent contract and Law 2.

### Bus 2 — Post Fallback Retry Queue (PERMISSIBLE exception)

```
post_queue (Supabase table)
  producers: routes/agents/publishing.js, engagement.js, ugc.js
  consumer: services/post-fallback.js (cron */5 * * * *)
  actions: publish_post, reply_comment, reply_dm, send_dm, repost_ugc
```

**Why permissible:** Serves user-initiated writes (human published content that failed), not autonomous acquisition. Exception is narrow — this is UX recovery, not orchestration.

### Bus 3 — Webhook + In-Memory Cache (frontend-facing, to be replaced)

```
Meta Webhook → /webhook/instagram → broadcastToFrontend() → global.realtimeCache (in-memory, 100-event cap)
  → GET /webhook/realtime-updates (frontend polling)
```

**Problems:** In-memory (lost on restart), 100-event cap, no persistence, frontend must poll.

### Bus 4 — Agent Heartbeat (Agent ↔ Frontend, keep)

```
Python Agent → POST /agent/heartbeat → agent_heartbeats table
  → GET /agent/status (frontend reads liveness from 25-min threshold)
```

### Bus 5 — Supabase Realtime (CONFIGURED BUT NOT USED — key opportunity)

```
lib/supabase/_client.js — realtime: { params: { eventsPerSecond: 10 } } ✓ CONFIGURED
lib/supabase/*.js      — NO .subscribe() calls anywhere in codebase
```

**Supabase Realtime is fully configured but completely unused.** This is the biggest gap in the architecture.

## 17. Architectural Laws (Updated)

These are repo-level invariants. Any code violating these laws is invalid by definition.

1. **Only the Supervisor may authorize execution.**
2. **Only the SyncLayer may authorize acquisition.**
3. **No graph may independently fetch external state.**
4. **Backend may never own orchestration semantics.**
